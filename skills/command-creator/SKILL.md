# Command Creator

## 概述

帮助用户创建 iFlow CLI 自定义命令，从命令定义到配置文件生成再到安装部署的完整流程。

## 创建流程

### 第一步：理解用户需求

询问用户以下信息以明确命令需求：

1. **命令用途**：这个命令是做什么的？
2. **触发场景**：用户会在什么情况下使用这个命令？
3. **功能描述**：命令应该执行什么具体操作？
4. **预期输出**：用户希望得到什么样的结果？

### 第二步：选择配置格式

根据用户需求选择合适的配置格式：

**TOML 格式** - 推荐用于复杂命令
- 结构清晰，支持更多元数据
- 适合需要详细描述的命令
- 格式：
```toml
description = "命令的简短描述"

prompt = """
命令的详细提示词内容
可以包含多行说明
"""
```

**Markdown 格式** - 适合简单命令
- 简洁直接，易于编写
- 适合单行或简单的提示词
- 格式：直接在 `.md` 文件中写入提示词内容

### 第三步：生成配置文件

根据选择的格式生成配置文件。

#### TOML 配置文件生成规则

1. **description 字段**：
   - 简短描述命令功能（1-2 句话）
   - 清晰说明命令的作用
   - 示例："专业代码审查工具，支持多语言代码质量检测"

2. **prompt 字段**：
   - 详细的命令执行指令
   - 使用三引号包裹，支持多行
   - 包含具体的操作步骤和期望输出

#### Markdown 配置文件生成规则

- 直接写入提示词内容
- 简单命令可以直接用单行
- 复杂命令可以使用多行

### 第四步：确定安装位置

询问用户安装到哪个作用域：

**全局作用域**
- 路径：`~/.iflow/commands/`
- 所有项目都可使用
- 适合通用工具命令

**项目作用域**
- 路径：`{project}/.iflow/commands/`
- 仅当前项目可用
- 适合项目特定命令
- 优先级高于全局命令

### 第五步：创建命令文件

根据选择的格式和位置创建配置文件。

#### 命令文件命名规范

- 使用小写字母和连字符
- 描述性名称，易于理解
- 示例：`code-reviewer.toml`, `security-scan.md`

## 命令文件结构示例

### 示例 1：代码审查命令 (TOML)

```toml
# Command: code-reviewer

description = "专业代码审查工具，支持多语言代码质量检测"

prompt = """
你是一个专业的代码审查专家。请分析用户提供的代码，从以下方面进行评估：

1. 代码质量和可读性
2. 安全性问题检测
3. 性能优化建议
4. 最佳实践遵循情况
5. 潜在的bug或逻辑错误

请提供具体的改进建议和示例代码。
"""
```

### 示例 2：安全检查命令 (Markdown)

```markdown
Review this code for security vulnerabilities:

- Check for common security issues
- Identify potential attack vectors
- Suggest security improvements
- Provide secure code examples
```

### 示例 3：文档生成命令 (TOML)

```toml
description = "智能文档生成工具，自动为代码生成详细文档"

prompt = """
请为提供的代码生成详细的文档，包括：

1. 函数/类的用途说明
2. 参数说明（类型、含义、默认值）
3. 返回值说明
4. 使用示例
5. 注意事项和最佳实践

文档应该清晰、准确、易于理解。
"""
```

## 命令分类参考

根据命令功能，建议选择合适的分类：

- **开发工具类**：refactor, implement, test, scaffold
- **代码质量类**：review, make-it-pretty, remove-comments
- **文档工具类**：docs, contributing, explain-like-senior
- **安全分析类**：security-scan
- **实用工具类**：commit, cleanproject, create-todos

## 验证命令

创建命令后，验证其有效性：

1. 检查配置文件语法是否正确
2. 确认 description 和 prompt 字段完整
3. 测试命令是否能正常加载
4. 验证命令输出是否符合预期

## 最佳实践

1. **描述要简洁明确**：description 应该让用户快速理解命令用途
2. **提示词要具体详细**：prompt 应该包含清晰的执行步骤
3. **使用有意义的命名**：命令文件名应该反映其功能
4. **合理选择作用域**：通用命令放全局，项目特定命令放项目级
5. **定期更新维护**：根据使用反馈优化命令

## 常见问题

**Q: 如何更新已存在的命令？**
A: 直接编辑对应的配置文件，修改后立即生效。

**Q: 可以同时使用 TOML 和 Markdown 格式吗？**
A: 不可以，一个命令只能使用一种格式。

**Q: 命令文件可以包含其他资源吗？**
A: 当前版本仅支持配置文件，不支持额外的资源文件。

**Q: 如何删除命令？**
A: 删除对应的配置文件即可。

## 命令管理命令

- `/commands online` - 浏览在线命令市场
- `/commands get <id>` - 查看命令详情
- `/commands add <id>` - 安装命令
- `/commands list` - 列出已安装命令
- `/commands remove <name>` - 删除命令
