# 规范优先开发模式

## 概述

规范优先开发模式是一种系统化的开发工作流，强调在编写代码之前先进行充分的规划、需求分析和技术设计。通过生成规范文档来确保开发方向的正确性，减少返工，提高代码质量。

**核心理念**："先规划，后执行"

## 触发条件

当用户提出以下类型的请求时，自动触发规范优先开发模式：

1. **新功能开发**
   - "我想要实现一个[功能]"
   - "添加一个[组件/页面]"
   - "开发[某功能]"

2. **Bug 修复**
   - "修复[某问题]"
   - "[某功能]有问题"
   - "[某功能]不工作"

3. **功能优化**
   - "优化[某功能]"
   - "改进[某功能]的性能"

## 自动产出

为每个功能或修复生成三个规范文档：

1. **需求文档** (`.iflow/specs/<feature-name>/requirements.md`)
   - 功能概述
   - 用户故事
   - 功能需求
   - UI/UX 需求
   - 数据需求
   - 技术约束
   - 依赖与排期
   - 风险评估

2. **设计文档** (`.iflow/specs/<feature-name>/design.md`)
   - 总体设计
   - 详细设计（组件、数据流、API、路由、国际化）
   - 类型定义
   - 实现细节
   - 测试策略
   - 代码规范

3. **任务清单** (`.iflow/specs/<feature-name>/tasks.md`)
   - 任务分解（多个阶段）
   - 每个任务的验收标准
   - 任务依赖关系
   - 进度跟踪

## 工作流程

### 阶段 1：需求分析

#### 1.1 理解用户需求
与用户深入沟通，明确：
- **功能目标**：这个功能要解决什么问题？
- **用户场景**：谁会使用？在什么场景下使用？
- **验收标准**：怎么判断功能是否完成？
- **优先级**：P0（必须有）/P1（应该有）/P2（可以有）
- **依赖关系**：是否依赖其他功能或资源？

#### 1.2 分析现有实现
在制定新计划前，必须：
1. 检查项目是否存在规范文档（如 `AGENTS.md`）
2. 查看相关功能的现有代码
3. 理解项目架构和代码规范
4. 识别可复用的组件和模式
5. 找出潜在的技术债务

#### 1.3 生成需求文档
创建 `requirements.md`，包含完整的需求定义。

### 阶段 2：技术设计

#### 2.1 架构设计
基于需求和现有代码分析，设计技术方案：

**设计原则**：
- 遵循项目现有架构（如果存在）
- 复用现有组件和工具
- 保持代码风格一致
- 考虑可扩展性和可维护性

#### 2.2 生成设计文档
创建 `design.md`，包含详细的技术设计。

### 阶段 3：任务分解

#### 3.1 将设计分解为可执行任务
基于技术设计，将功能分解为具体的、可执行的任务。

#### 3.2 生成任务清单
创建 `tasks.md`，包含详细的任务分解。

### 阶段 4：用户确认

向用户展示生成的规范文档，等待用户确认或修改意见。

### 阶段 5：执行开发（可选）

如果用户确认后要求开始开发：
1. 按照 `tasks.md` 执行任务
2. 逐步完成每个阶段
3. 更新任务状态

## 学习与优化机制

### 反馈收集
在执行过程中，收集以下反馈：

1. **用户纠正**
   - 需求理解错误
   - 技术方案不合理
   - 任务分解不准确

2. **项目规范学习**
   - 代码风格习惯
   - 组件命名规范
   - API 设计模式

3. **最佳实践发现**
   - 项目特有的实现方式
   - 常用的工具函数
   - 常见的设计模式

### 技能更新
根据收集的反馈，定期更新技能：

1. **更新需求模板**
   - 添加常见的字段
   - 优化问题引导
   - 增强验收标准

2. **更新设计模板**
   - 添加常见的设计模式
   - 优化架构描述
   - 补充最佳实践

3. **更新任务模板**
   - 添加常见的开发任务
   - 优化任务分解
   - 增强检查清单

4. **更新项目知识库**
   - 记录项目特有的规范
   - 记录常用的组件和工具
   - 记录常见问题和解决方案

### 知识沉淀
将学习到的知识沉淀到以下位置：

1. **SKILL.md** - 技能本身
2. **references/** - 参考文档
3. **AGENTS.md** - 项目上下文（如果适用）

## 通用规范应用

每次执行前，检查并应用项目的通用规范：

1. **项目配置检查**
   - TypeScript 配置（`tsconfig.json`）
   - ESLint 配置（`eslint.config.js` 或 `.eslintrc`）
   - Prettier 配置（`.prettierrc`）
   - 其他配置文件

2. **代码规范检查**
   - 导入顺序规范
   - 文件命名规范
   - 组件结构规范

3. **项目结构检查**
   - 了解项目的目录结构
   - 识别可复用的组件和工具
   - 理解状态管理方案

## 注意事项

1. **不要跳过步骤**：必须按照工作流程执行，不能直接开始编码
2. **用户确认优先**：在开始编码前，必须获得用户对规范文档的确认
3. **记录一切**：记录所有用户反馈和纠正，用于优化技能
4. **保持更新**：定期根据项目变化更新技能
5. **灵活调整**：根据任务复杂度调整文档详细程度
6. **适应性**：根据项目是否存在规范文档，动态调整工作流程

## 输出格式

所有生成的文档都使用中文编写，确保清晰易懂。

## 快速参考

### 文件路径规范
- 需求文档：`.iflow/specs/<feature-name>/requirements.md`
- 设计文档：`.iflow/specs/<feature-name>/design.md`
- 任务清单：`.iflow/specs/<feature-name>/tasks.md`

### 命名规范
- feature-name：kebab-case（如 `shopping-cart`）
- ComponentName：PascalCase（如 `ShoppingCart`）
- functionName：camelCase（如 `calculateTotal`）

### 状态标记
- 📋 待开始
- 🟡 进行中
- 🟢 已完成
- ⚠️ 有问题

## 常见问题

**Q: 必须生成所有三个文档吗？**
A: 对于复杂功能，建议生成所有文档。对于简单修复，可以简化文档内容。

**Q: 如果项目没有规范文档怎么办？**
A: 使用通用的软件开发最佳实践，并在文档中注明。

**Q: 可以跳过某些阶段吗？**
A: 不建议跳过，但可以根据任务复杂度调整各阶段的详细程度。

**Q: 如何跟踪任务进度？**
A: 使用 tasks.md 中的进度跟踪表格，实时更新任务状态。
